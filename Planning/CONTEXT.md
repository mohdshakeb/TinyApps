# Planning Context — TinyApps

## What Is TinyApps

TinyApps is a container for multiple small, self-contained web apps and experiments — client demos, one-off tools, quick builds. Each app lives in its own subfolder under `src/`; there's no shared framework forced across them. The first app being built is **Wildstone Scent Aura**, a web AR client-demo experience for the Wildstone deodorant brand.

## Tech Stack

- **Framework:** Per-app choice. Wildstone Scent Aura: Vanilla HTML5/CSS3/JS + Canvas/WebGL — no heavy framework, to keep it demo-fast and light.
- **Target:** Mobile-first responsive web, running on the client's own device (unknown make/model) — treat as broad-compatibility, not a single controlled device.
- **Storage:** None for demo apps — client-side only, no backend.
- **Deploy:** Vercel, per app, via a GitHub repo connected through Vercel's git integration (Root Directory set per-app, e.g. `src/WildstoneScentAura`).

## Architectural Principles

- Each app is self-contained under `src/<AppName>/` — no cross-app shared framework or build system unless a real need emerges.
- Prefer vanilla browser-native tech over heavy frameworks to keep apps quick to build and light to load.
- Client-demo apps default to the PRD's stated performance/compatibility budget (not a relaxed one) whenever the demo device isn't controlled by us — see Wildstone decisions below.
- Spike the highest-risk technical unknown before building the full feature around it (e.g., test capture/recording APIs on target browsers before investing in the surrounding UI).

## MVP Features

_Wildstone Scent Aura (first app in this collection):_

1. **Camera-based aura visual** — Lightweight face-detection-driven WebGL/Canvas particle aura, Wildstone Edge variant only for the demo.
2. **Capture** — Snapshot capture (`canvas.toDataURL`) confirmed reliable; video capture (`MediaRecorder`) only if the iOS Safari spike proves it works.
3. **Touch Canvas fallback** — Minimal-fidelity interactive fallback if camera access fails live during the demo.

## Architecture Decisions

### 2026-07-05 — Face tracking: lightweight face detection, not full mesh
**Decision:** Use a lightweight face-detection model (bounding box/landmarks) rather than a full face-mesh model for the aura's tracking input.
**Rationale:** Real tracking gives the "wow factor" a client demo needs (more than cursor/touch-driven fake tracking), but full mesh tracking is heavy on bundle size and per-frame compute. A lighter detection model is the middle ground given the demo runs on the client's own (unknown, uncontrolled) device.

### 2026-07-05 — Device target: client's own device, not a controlled one
**Decision:** Keep the original PRD performance budget (<3MB initial load, 60fps target on mid-range Android/iOS Safari) as a real constraint rather than relaxing it.
**Rationale:** The demo will run on whatever device the client picks up, not a device we rehearse on ahead of time. This is effectively the same compatibility bar as a public launch, so we can't cut corners on performance the way we could for a single controlled device.

### 2026-07-05 — Variant scope: Wildstone Edge only for v1
**Decision:** Build and fully polish only the Wildstone Edge variant (electric blue lightning/energy arcs) for the demo. Red and Intense Black are fast-follows after the demo.
**Rationale:** Narrowing scope to one variant protects build time and polish quality given there's no fixed deadline yet but the work should still be tightly scoped.

### 2026-07-05 — Capture: spike iOS MediaRecorder first, snapshot as safe fallback
**Decision:** Spike `MediaRecorder` canvas-capture on iOS Safari before committing to video capture as a demo feature. If unreliable, ship snapshot-only capture (`canvas.toDataURL`) for the demo.
**Rationale:** iOS Safari's support for capturing a composited canvas+video stream has a history of inconsistency. A capture failure live in front of a client is the worst-case failure mode, so this gets tested first, before any aura visuals are built.

### 2026-07-05 — Analytics: out of scope for demo
**Decision:** No engagement/share analytics instrumentation for the demo build.
**Rationale:** The audience is one client in a room, not a public campaign. Revisit if this moves from demo to a real public rollout.

### 2026-07-05 — Touch Canvas fallback: build as demo insurance
**Decision:** Build the Touch Canvas fallback (finger-drag aura on a gradient card) even though it doubles the interactive systems to build, and keep it minimal-fidelity rather than full parity with the camera mode.
**Rationale:** Because the demo runs on an unrehearsed client device, camera permission/hardware failure is a real live risk. Having a fallback prevents the whole demo from going dark if the camera doesn't init.

### 2026-07-05 — Build tooling: Vite vanilla-JS, not bundler-free
**Decision:** Use Vite's vanilla-JS template (no TypeScript, no framework, no Three.js) instead of plain unbundled static files.
**Rationale:** `@mediapipe/tasks-vision` is a bare-specifier npm import; without a bundler it would need to load from a third-party CDN at demo time, an unacceptable live-demo risk on an unrehearsed device. Vite's dynamic-import code-splitting also lets the multi-MB MediaPipe WASM+model chunk lazy-load after the initial splash paint, reconciling the PRD's <3MB shell budget with a genuinely heavy dependency. Vite auto-detects on Vercel with zero config.

### 2026-07-05 — Self-host MediaPipe WASM/model assets, don't load from CDN
**Decision:** Download and commit MediaPipe's WASM runtime and `blaze_face_short_range.tflite` model into `public/mediapipe/` rather than loading them from a CDN at runtime.
**Rationale:** Same live-demo-reliability reasoning as the bundler choice — a third-party CDN in the critical path is a risk we don't need to take when self-hosting is straightforward.

### 2026-07-05 — Aura renderer: Canvas2D primary, WebGL as optional stretch
**Decision:** Build the Wildstone Edge particle aura on Canvas2D as the committed path; only attempt a WebGL shader backend later if perf headroom allows, behind a swappable `AuraRenderer` seam.
**Rationale:** Canvas2D is lower-risk across an unknown client GPU/driver than raw WebGL shaders, and the device is explicitly unrehearsed.

### 2026-07-05 — Shared anchor interface for camera and Touch Canvas input
**Decision:** Both the face-tracker and the Touch Canvas fallback emit the same `{ x, y, scale, confidence }` shape; the aura/particle system consumes only this interface and never knows which input source is live.
**Rationale:** Means one aura system serves both the camera path and the fallback path, instead of building and maintaining two — meaningfully less work over a 1-month timeline.

### 2026-07-05 — Testing strategy: small Playwright + node:test suite, real-device testing for the rest
**Decision:** Automate only deterministic/mockable logic (state machine transitions, permission-denied → fallback branch, share-sheet trigger, a fake-media-device happy-path smoke test) — 4-6 tests total. Do not attempt to automate aura visual quality, real face-tracking accuracy, or iOS Safari MediaRecorder reliability.
**Rationale:** Playwright's `webkit` project is desktop WebKit, not real mobile iOS Safari — it will not reproduce the WebKit bugs the video-capture spike is specifically worried about. A green automated suite must not be mistaken for real-device validation; those risk areas require hands-on testing on the real iPhone/Android devices available for this project.

### 2026-07-05 — Deploy pipeline: git repo → GitHub → Vercel git integration
**Decision:** Initialize git in `TinyApps/`, push to a new GitHub repo, connect it to Vercel with Root Directory `src/WildstoneScentAura` and the Vite framework preset (auto-detected).
**Rationale:** Gives an automatic preview URL per push, which the session plan relies on for weekly real-device testing, over the alternative of manual `vercel deploy` CLI pushes with no git history.

### 2026-07-05 — Naming conventions: PascalCase for screens/renderers, camelCase for logic/utils
**Decision:** PascalCase for component-like/screen files (`SplashScreen.js`, `AuraRenderer.js`), camelCase for logic/utility files (`faceTracker.js`, `cameraStream.js`), lowercase plain folders for categories (`screens/`, `camera/`, `aura/`).
**Rationale:** No existing local pattern covers this app type (vanilla JS + Canvas, no framework); this aligns with the closest workspace precedent (Focal/GmmcoWorkshop's Vite projects) rather than inventing an unrelated convention.

### 2026-07-05 — Session 0 finding: self-hosted MediaPipe payload is ~22MB
**Decision:** Confirmed via actual build output (not estimate): self-hosting `@mediapipe/tasks-vision`'s WASM runtime (both SIMD and non-SIMD variants, since the correct one isn't known until runtime) plus the `blaze_face_short_range.tflite` model totals ~22MB. The initial page shell (HTML/CSS/JS, no tracker) is ~138KB uncompressed — well under budget on its own.
**Rationale/implication:** This confirms and sharpens the earlier "<3MB is the initial-shell budget, not the whole app" decision — the gap between shell and tracker is much larger than a rough estimate suggested. Reinforces that `faceTracker.js` must be dynamically imported only after the user taps past the splash screen (planned for Session 3), never eagerly bundled. Full detail in `src/WildstoneScentAura/PROGRESS.md` (Session 0).

### 2026-07-05 — Session 2 (Android) finding: main-thread detection blocks the frame rate, Worker offload built now
**Decision:** Real-device measurement (Android, camera feed only, no aura yet) showed `detectForVideo()` blocking the main thread for 20-35ms per call at the ~15Hz throttle rate — roughly 300-500ms of blocking per second, causing main-thread FPS to swing between 30 and 57 even with nothing else running. Built the Worker/OffscreenCanvas split (`faceTrackerWorker.js`) immediately rather than deferring it, moving `FaceDetector` init and `detectForVideo` entirely off the main thread; frames are captured via `createImageBitmap(videoEl)` and transferred to the worker per detection tick.
**Rationale:** This was flagged as conditional architecture in the original plan ("only if perf data shows main-thread contention") — this data point is exactly that trigger. Building it now, before Session 3's state machine and Session 4's aura renderer are built on top, avoids reworking that code later once a 60fps particle system would have made the stutter directly visible and harder to isolate. Side benefit confirmed in the build output: moving the `@mediapipe/tasks-vision` import into the worker dropped the main entry chunk from ~139KB to ~3.5KB, since the library now bundles into the lazy-loaded worker chunk instead of the main thread's JS.

### 2026-07-05 — Session 2 (Android) CONFIRMED: Worker offload fixes main-thread FPS; worker must run classic, not module
**Decision:** First real-device retest of the Worker offload surfaced two bugs, both fixed: (1) `faceTrackerWorker.js` was instantiated with `{ type: 'module' }`, but `@mediapipe/tasks-vision` 0.10.35 loads its WASM glue via `importScripts()`, which throws inside a module worker (disallowed by spec) — the library's fallback (dynamic `import()`) runs in its own module scope, so the glue's `ModuleFactory` global never got set, throwing `Error: ModuleFactory not set` at init. Fixed by dropping `{ type: 'module' }` so the worker runs classic; Vite's default `worker.format: 'iife'` already bundles the worker's ESM imports into a self-contained script, so no `vite.config` changes were needed. (2) `debugOverlay.js` was `JSON.stringify`-ing `Error` objects for its on-screen console mirror, which always yields `"{}"` (Error properties are non-enumerable) — this was hiding the real error text during diagnosis. Fixed with an `instanceof Error` special case.
**Result:** With both fixed, Chrome on Android confirms `mainThreadFps` holds steady at **60** with the Worker offload live, `avgDetectLatency` fluctuating ~20-31ms (expected — no longer blocks the main thread). **Session 2 (Android half) is done — Worker offload is confirmed sufficient.**
**Rationale/implication:** The Worker offload built in the prior entry had never actually been exercised on a real device before this — it was new code, not a regression. Brave on the same device separately denies camera permission with no prompt (a persisted site-permission/Shields state, not a code issue); Chrome's confirmation is sufficient to unblock Session 3. iPhone half of Session 2, and all of Session 1, remain deferred until an iPhone is available (per the Session 0 sequencing change) — nothing else in the plan is blocked by this.

### 2026-07-05 — Session 3: state machine + screen shell built, main.js becomes a pure composition root
**Decision:** Implemented `appStateMachine.js` as the full state graph from `Planning/WildstoneScentAura_Plan.md` (all 7 states, not just the ones with screens built so far) since the transition table is cheap and the graph shape is already fixed by prior planning — no speculative states were added beyond what's documented. `main.js` was rewritten from the old linear `boot()` function into a composition root that mounts/unmounts a screen module per FSM state and feeds async results (camera permission, tracker init) back in via `machine.send(...)`. `TouchCanvasFallbackScreen.js` is a placeholder for now — the real finger-drag interaction is deliberately deferred to Session 4, per the plan's shared-anchor-interface decision above.
**Rationale:** Keeps the FSM itself trivially testable in isolation (`node:test`, no DOM) while screens stay dumb (mount/unmount + a couple of callbacks), matching the plan's testing-strategy decision to automate only deterministic logic. Wiring the camera track's `ended` event to `CAMERA_LOST` was added here too (one line, directly in the documented state graph) so a camera dying mid-demo now degrades gracefully instead of freezing on the last frame — worth doing now since Session 4/5 will build on top of this shell and a frozen-camera bug would be much harder to isolate once the aura renderer is layered on.
**Real-device confirmation (Android):** Splash → Start → Live Aura navigates correctly with the debug bounding-box overlay drawing accurately over the live feed; the forced-deny path also confirmed landing cleanly on the Touch Canvas Fallback placeholder. **Session 3 is done** on Android; iPhone confirmation remains part of the standing deferred-until-hardware-available gap from Sessions 1-2.

### 2026-07-05 — Session 4: Canvas2D aura v1 built (particleSystem + EdgeVariant + AuraRenderer + touchAuraController)
**Decision:** Built the aura along the exact seam named in the Session 4 plan: `particleSystem.js` is fully variant-agnostic (owns the pool + velocity/drag/life physics, delegates only particle creation and drawing to a variant object) so Red/Intense Black variants can reuse it unchanged later; `EdgeVariant.js` supplies the Wildstone Edge look (jagged electric-blue arc fragments); `AuraRenderer.js` runs its own rAF loop independent of the ~15Hz face-detection rate and lerps toward each new anchor so motion stays smooth at 60fps between throttled updates; `touchAuraController.js` drives the identical `{x,y,scale,confidence}` anchor shape from Pointer Events, so the Touch Canvas fallback reuses the exact same renderer/variant as the camera path with zero aura-specific code duplication. Also built `src/utils/coverCrop.js` for exact `object-fit: cover` + mirror math, replacing the approximate mapping Session 3 flagged as a carried-forward gap, since the aura anchor (unlike the old debug overlay) needs to track the face precisely rather than approximately.
**Rationale:** This is the first session where a design choice from several sessions ago (the shared anchor interface, the swappable `AuraRenderer` seam) gets cashed in — both input paths converging on one renderer is exactly the reduced-maintenance-surface bet made back in Session 0/3 planning. Chose `globalCompositeOperation = 'lighter'` for the glow instead of per-particle `ctx.shadowBlur` specifically because of Session 2's real-device lesson that main-thread Canvas work is not free on the class of device this demo targets — `shadowBlur` is a known-expensive Canvas2D op on mobile, and this demo now runs it every rAF frame across potentially 100+ particles, so the cost avoidance is deliberate, not premature optimization.
**Not yet real-device confirmed:** built and verified via `npm run build`, the existing unit/e2e suite (unchanged, still green), and an ad-hoc Playwright pixel-read smoke check (drag → non-transparent canvas pixels), but **not yet tested on an actual phone**. The two things that specifically need real-device eyes before Session 4 can be marked done per the plan's own criteria: (1) whether main-thread frame rate holds up now that a 60fps particle loop runs alongside the Worker-offloaded face detector — this is the exact scenario Session 2's CONTEXT.md entry warned would make any main-thread stutter "directly visible and harder to isolate"; (2) whether the aura's apparent size (`REFERENCE_FACE_FRACTION = 0.35` in `faceAnchor.js`, clamped [0.7, 1.6]) actually looks right relative to a real face at typical selfie distance — this constant is a guess, not a measurement.

### 2026-07-05 — Session 4 real-device retest (Android): perf confirmed, aura repositioned from on-face to around-face
**Decision:** First real-device retest confirmed (1) of the two open perf/visual questions above — `mainThreadFps` holds at 60 with the particle loop running alongside the Worker-offloaded detector, so no main-thread regression from combining them. It also surfaced a visual bug not caught by any automated check (expected, per the plan's testing-strategy decision that aura visual quality is explicitly not automated): the aura rendered directly **on** the face rather than around it, because `EdgeVariant.createParticle` spawned every particle at the anchor's exact center point. Fixed by spawning particles on a ring around the anchor instead (`RING_RADIUS_FRACTION = 0.18` of `min(canvas width, height)`, scaled by `anchor.scale`), with the jagged arc continuing radially outward from the ring rather than from dead-center; threaded `width`/`height` through `particleSystem.js` → `variant.createParticle(...)` so a variant can size a spawn radius relative to the canvas.
**Rationale:** Matches the PRD's own description of the aura as emitting "outwards behind or around the coordinate center," not on top of it — spawning at dead-center was a literal reading of "at the anchor" that didn't account for the face itself occupying that same point. This is exactly the class of issue the project's testing strategy predicted would only surface on a real device/real face, not from Canvas2D code review alone.
**Still open:** the ring-spawn fix itself hasn't been re-confirmed on device yet (pushed, not yet retested) — next step is confirming the halo now frames the face at a reasonable radius, then Session 4 can be marked done.

**Retest confirmed the ring-spawn fix — Session 4 is DONE on Android.** The aura frames around the face rather than sitting on it; current tuning (`RING_RADIUS_FRACTION = 0.18`, `REFERENCE_FACE_FRACTION = 0.35`) is acceptable as-is. Further visual tuning, if wanted later, belongs to Session 7 ("Visual polish + WebGL stretch decision") rather than reopening this session. iPhone confirmation remains part of the standing deferred-until-hardware-available gap from Sessions 1-2.

### 2026-07-05 — Session 5: capture + share built snapshot-only; state machine gains origin-tracked RETRY/DONE

**Decision:** Built `snapshotCapture.js` (composite video+aura canvas via `canvas.toBlob`), `BrandOverlay.js` watermark, and `shareSheet.js` (`navigator.share` first, download-link fallback), per the plan's Session 5 scope. `videoCapture.js` was **not** built — Session 1's iOS `MediaRecorder` spike still hasn't run (no iPhone available), so per the plan's own explicit fallback rule ("otherwise snapshot-only"), `captureFeatureFlags.js` ships with `ENABLE_VIDEO_CAPTURE = false` and video capture stays deferred rather than being built speculatively. Also changed `appStateMachine.js`: `CAPTURE_PREVIEW`'s `RETRY` and `SHARE`'s `DONE` used to point unconditionally at `LIVE_AURA` (as literally drawn in this plan's own state graph), which breaks the Touch-Canvas-fallback branch — a camera-denied user who captures from the fallback screen would get bounced into a state needing a camera they don't have. Added a single `captureOrigin` memory cell (set on the `CAPTURE` event, consumed by a `RETURN_TO_ORIGIN` sentinel) so both `RETRY` and `DONE` return to whichever branch (`LIVE_AURA` or `FALLBACK_TOUCH_CANVAS`) the capture actually started from.

The fix is a generic one-cell "return to origin" mechanism, not fallback-specific branching, so it costs one Symbol + one Set and no new states.

### 2026-07-05 — Post-Session 7 pivot: aura becomes a scored shake-to-release game, Edge's arc look replaced by vapor
**Decision:** Reframed the aura from a continuous, always-on particle trail (emitting any time a face is detected, regardless of movement) into a mini-game: particles now emit only during an active head-shake motion, scored on amplitude + tempo-consistency (not raw speed) into too-slow/too-fast/perfect, landing on a retry-only results screen. `EdgeVariant.js`'s jagged electric-blue lightning-arc look is fully replaced (not kept as an alternate) by `VaporVariant.js` — soft floating mist/dot particles whose drift and coherence scale with shake energy. Capture/share is explicitly unchanged: no slow-mo or score-burned share video (considered and dropped), and the results screen has no direct capture path — capture still happens via the existing button back in `LIVE_AURA`. Inserted as Session 8 in `Planning/WildstoneScentAura_Plan.md`, renumbering the remaining not-yet-started sessions (Bundle budget, Edge-case hardening, Full demo rehearsal) to 9-11.
**Rationale:** Came out of a brainstorming pass on making the demo more memorable/shareable than a passive ambient effect. Shaking to "release" the aura mirrors the literal act of shaking a deodorant can, and scoring the gesture gives users a reason to retry — both directly address the discoverability/engagement gap a purely ambient aura has in a one-off client demo. The motion-energy gate is deliberately generic (lives in `AuraRenderer.js`, the one place cross-frame anchor state already exists) so it benefits the Touch Canvas fallback path for free, while the scoring logic (`shakeTracker.js`) is kept `LIVE_AURA`-specific since it's game logic, not rendering. Slow-mo/score-burned video was dropped specifically because it would have reopened new capture-pipeline risk on top of the still-unresolved iPhone `MediaRecorder` gap (Sessions 1/2/5) — not worth compounding an already-open risk for a nice-to-have.

**Built:** `npm run build`, `npm run test:unit` (11/11), and `npx playwright test` (2/2, existing specs unmodified) all green; an ad-hoc Playwright pixel-read smoke check confirmed the core property end-to-end in a real browser engine — zero particles while a pointer is held still, particles appear once there's actual motion. **Not yet real-device confirmed** — `shakeTracker.js`'s scoring constants and `AuraRenderer.js`'s motion-energy constants are explicit starting estimates against a real face-tracked shake, same gap pattern as every prior visual-tuning session. Full detail in `src/WildstoneScentAura/PROGRESS.md` (Session 8).

### 2026-07-06 — Session 9: scored aura release flow replaces the shake-game/results-screen split, client feedback on the released particle look

**Decision:** Collapsed Session 8's two disconnected mechanics (shake-scoring landing on a standalone `ShakeResultsScreen.js`, and a separate always-on manual tap/hold capture button) into one guided flow. `LIVE_AURA` gained an idle sub-state — camera + face tracking live, particle emission and `shakeTracker` both gated off, a single "Release your aura" button shown. Tapping it arms everything at once: opens the particle gate, starts `shakeTracker`, and auto-starts a fixed 10-second `videoCapture.js` recording (`MAX_DURATION_MS` 5000 → 10000, no more long-press gesture — this is the sole capture trigger now). `shakeTracker.js`'s `MAX_ROUND_MS` extended 4000 → 9500 to fit the clip; once a round completes the score locks for the rest of the clip. `compositeFrame.js` gained a score+bucket overlay burned directly into the video alongside the existing watermark — **this explicitly reverses Session 8's "no slow-mo or score-burned video" call above**, once video-burned scoring was actually discussed with the client and preferred over a standalone results screen. `ShakeResultsScreen.js` and the `SHAKE_RESULTS`/`SHAKE_COMPLETE` FSM state/event are removed entirely. Separately, the client reviewed the released-particle look against a reference image and asked for two changes: `VaporVariant.js`'s palette moved from pale near-white/cyan "bubble" tones to a saturated multi-hue sparkle palette with a brighter core, and the spawn origin widened from a tight ring on the face to a region offset around/below the face anchor ("around the person," not "on the face" — still no body/pose tracking, a distribution estimate only off face position/size). Particles also stopped dying on a short `maxLife` — they now hand off to an idle floating/bobbing phase and persist for the run of the capture, with `particleSystem.js` gaining an oldest-first eviction cap (`maxParticles` raised to 400) since particle death no longer frees pool slots on its own.

**Rationale:** The Session 8 split meant the score (the whole point of the mini-game) and the capture button (the whole point of the demo) were disconnected — a good shake produced a number nobody could keep, and a good capture had no score. Burning the score into the video itself resolves that with no new capture-pipeline risk beyond what `videoCapture.js` already carries (same composite-and-record pipeline as `snapshotCapture.js`, which was consequently deleted as dead code once the manual capture button it served was removed). The particle-look changes came directly from the client comparing the shipped look against a reference image mid-review, not from a technical constraint.

**Built:** `npm run test:unit` (9/9, two obsolete `SHAKE_RESULTS` cases removed), `npx playwright test` (2/2, unmodified Touch Canvas specs still pass), `npm run build` all green. **Not yet real-device confirmed** at the time this was built — see the Session 10 entry immediately below, which is exactly that real-device retest surfacing three further feel problems. Full detail in `src/WildstoneScentAura/PROGRESS.md` (Session 9).

### 2026-07-06 — Session 10: aura feel fixes from real-device client feedback

**Decision:** Real-device testing of Session 9 surfaced three feel problems, now scoped as Session 10: (1) particles don't track head movement once they've settled into their floating idle phase — they orbit a permanently frozen point instead of drifting with the person; (2) the release still reads as bubbles popping in randomly rather than "water shaking off after a bath" — the spawn area is far too large/diffuse and there's no gravity or directional flick to the burst; (3) the score only appears for the last fraction of a second of the 10-second clip, since it's only computed once the shake round locks, and locking is usually triggered by the 9.5s safety cap rather than an earlier pause. Fix approach: `particleSystem.js` gets a delta-based "leash" follow on settled particles (gated on the same confidence-trust threshold as emission, so it can't chase Touch Canvas's release-to-center neutral anchor) plus a variant-supplied gravity term on burst-phase particles; `VaporVariant.js`'s spawn area shrinks to a tight region around the head and its burst velocity becomes a gravity-affected spray-cone instead of a soft lerp; `shakeTracker.js` drops its push-based `onRoundComplete` callback for a pull-based `getScore()` so the score display updates live from when the shake begins instead of revealing only at lock (verdict/bucket text still waits for lock; only the number is live). Full design, root-cause analysis, and exact starting-estimate constants in `Planning/Session10_AuraFeelFixes_Plan.md`.

**Rationale:** Post-processing the video to burn in the score after recording (instead of live-updating during capture) was considered and explicitly ruled out with the client — it would need either a WebCodecs dependency with materially riskier iOS support than this project's already-shaky `MediaRecorder` path, or roughly doubling the capture wait via a second real-time re-encode pass. Live-updating stays inside the existing composite-and-record architecture at no extra cost. None of `particleSystem.js`/`VaporVariant.js`/`shakeTracker.js`'s internals have automated test coverage, so this is unconstrained refactoring, not a breaking API change to guard.

**Built:** `npm run test:unit` (9/9, unchanged), `npm run build`, `npx playwright test` (2/2, unmodified specs) all green. **Not yet real-device confirmed** — `FOLLOW_GAIN`, `GRAVITY_PX_S2`, `SPRAY_SPEED_MIN/MAX`, `SPRAY_HALF_ANGLE_RAD`, and the tightened spawn-ellipse ranges are all explicit starting estimates against a real face-tracked shake, same gap pattern as every prior visual-tuning session. Full detail in `src/WildstoneScentAura/PROGRESS.md` (Session 10).

### 2026-07-06 — Session 10.1/10.2: real-device retest of Session 10 surfaced 3 more issues

**Decision:** The client's first real-device pass on Session 10 surfaced three more problems. (1) Particles fell straight down like dropped balls instead of scattering — root cause was the burst angle being centered on the live shake direction (collapsing a left-right head shake into two narrow horizontal cones) combined with `GRAVITY_PX_S2` outlasting the initial burst speed. Fixed by making the burst angle fully random per particle (true 360° scatter) and removing gravity entirely — it was speculative variant-agnostic infrastructure in `particleSystem.js` with only one consumer, which no longer needed it. (2) Settled particles still felt independent of head movement — root cause was the Session 10 follow-nudge reading `AuraRenderer.js`'s `smoothedAnchor`, the same render-lerped value whose dilution was already diagnosed and fixed for the motion-energy signal between Sessions 8-9 (a real detection only moves `smoothedAnchor` ~22% of the way per 60fps frame). Fixed by moving the follow-delta accumulation into `AuraRenderer.js`'s `setAnchor`, measured from raw anchor arrivals exactly like the existing motion-energy fix, then drained into `particleSystem.update()` once per frame. (3) The released look didn't match the client's reference image (`docs/references/image.png`) — a sparse, mostly warm-white pinpoint-sparkle field, not the dense, evenly-saturated, blob-like field Sessions 9-10 built. `VaporVariant.js`'s palette, particle size distribution, glow falloff, spawn spread, and density (`emissionRate`, `maxParticles`) were all reworked to match.

**Rationale:** All three were diagnosed by re-reading the actual code against the reported symptoms (not guessed at) — the first two are straightforward bugs (a physics combination and a stale-signal reuse of an already-documented trap), not new design decisions; the third is a deliberate look pass against a client-supplied reference. None of the affected files have automated test coverage, so this was unconstrained refactoring.

**Built:** `npm run test:unit` (9/9), `npm run build`, `npx playwright test` (2/2) all green after both fix rounds. An ad-hoc Playwright smoke check (Touch Canvas fallback, synthetic drag, screenshot; discarded after use) showed a sparse scatter of small mixed-color points consistent with the intended look. **Not yet real-device confirmed** — same gap as every prior visual-tuning session; this is the primary open item before Session 11. Full detail in `src/WildstoneScentAura/PROGRESS.md` (Session 10.1/10.2).
